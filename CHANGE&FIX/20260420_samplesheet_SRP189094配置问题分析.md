# 20260420 SRP189094 Ribo-seq 分析故障诊断与配置优化记录

## 1. 初始问题现象
在执行 nf-core/riboseq 工作流时，流程在 ORF 预测阶段（RiboTricer / RiboTISH）产出了空文件，或在任务中直接报 `FAILED 0`。

## 2. 问题排查与确诊
通过检查底层的 `.command.err` 日志和 MultiQC 预处理报告，得出以下结论：

1. **数据量没问题**：SortMeRNA 成功去除了 30%~47% 的 rRNA，剩余的 `non_rRNA` reads 高达 3980万~7070万，绝对数据量非常充足。
2. **读长分布异常（核心原因）**：高质量 Ribo-seq 的核糖体保护片段（footprints）应集中在 28-32 nt。但这批数据在比对后的主峰却在 **49-50 nt**（单峰达几十万条）。这表明实验中 RNase 消化很不充分，包含了较多过长片段（双核糖体或未消化的 mRNA 碎片）。
3. **三核苷酸周期性丧失**：长读段破坏了三核苷酸周期性（Periodicity）。RiboWaltz 无法准确估算 P-site 偏移量，RiboTricer 报错 `WARNING: no periodic read length found`。由于没有明显的 Frame 0 富集，导致严格依赖周期性的 ORF 预测算法（RiboTricer/RiboTISH）全军覆没。

## 3. 失败的抢救尝试
最初尝试的解决方案：
- **操作**：修改了 `riboseq.config`，对 `TrimGalore` 强加了 `--length 28 --max_length 35` 参数，试图过滤掉长读段，只保留潜在的单核糖体 footprints；同时新建了仅包含最佳样本对（Pair 4）的 `samplesheet_rescue.csv`。
- **结果**：工作流重新运行后，RiboTricer 和 RiboWaltz 依然报错退出（`exit status 1`）。
- **结论**：这批数据即使严格限制在 28-35nt 区间，剩下的 reads 依然**完全不具备**三核苷酸周期性规律。数据在建库层面的周期性特征已无法恢复。

## 4. 最终解决策略：放弃 ORF 预测，保留翻译效率（TE）分析
既然数据彻底不具备周期性，强行进行 ORF 预测已无意义。但数据仍可用于基因层面的表达定量和翻译效率（Translational Efficiency）分析。Anota2seq 等模块对周期性的要求较低，只需 CDS 区有 Count 即可。

### 最终操作记录：
1. **撤销了读长过滤限制**：在 `riboseq.config` 中移除了 `--length 28 --max_length 35`，允许全长片段进入定量环节。
2. **彻底禁用周期性依赖模块**：在 `riboseq.config` 中设置：
   ```groovy
   skip_ribotricer = true
   skip_ribotish = true
   skip_ribocode = true
   skip_ribowaltz = true
   ```
3. **保留翻译效率分析**：保持 `skip_anota2seq = false`。
4. **恢复原始样本表**：删除了抢救用的 `samplesheet_rescue.csv`，将 `riboseq.config` 中的 `input` 重新指向全量原始样本表 `samplesheet_SRP189094.csv`。

## 5. 后续运行建议
配置修改完成后，数据已被当做“富集了 CDS 的普通 RNA-seq”对待，可以顺利绕过所有报错节点，最终产出基于 Anota2seq 的组间翻译效率对比结果。
